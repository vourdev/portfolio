BAB II — LANDASAN TEORI

2.1 Clean Code, SOLID, DRY & KISS

Martin (2008) mendefinisikan clean code sebagai kode yang mudah dibaca manusia. Prinsip SOLID (Martin, 2017) terdiri dari Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, dan Dependency Inversion. DRY (Hunt & Thomas, 1999) menekankan setiap pengetahuan harus memiliki satu representasi otoritatif, sementara KISS mendorong solusi paling sederhana.

2.2 Debugging dan Logging

Debugging adalah aktivitas sistematis untuk menemukan, mengisolasi, dan memperbaiki kesalahan perilaku perangkat lunak (McConnell, 2004). Strateginya meliputi reproduksi bug, pembacaan stack trace, logging terstruktur, serta tail logs secara real-time. Laravel menyediakan Log facade, Whoops, dan paket laravel/pail untuk tailing log.

2.3 Dokumentasi Kode (PHPDoc & JSDoc)

Dokumentasi inline (PHPDoc untuk PHP, JSDoc untuk JS/TS) memberikan contract tertulis tentang parameter, tipe kembalian, dan efek samping fungsi. Standar PSR-5 mengatur format docblock PHP. Komentar konfigurasi pada file config/*.php Laravel juga merupakan bentuk dokumentasi self-explanatory yang sangat membantu pemeliharaan.

2.4 Spesifikasi Program & Type System

Spesifikasi program adalah deskripsi formal tentang apa yang harus dilakukan sistem. Pada level kode, spesifikasi diekspresikan melalui type hints (PHP 8 union/return types), TypeScript di frontend, skema validasi (Laravel validate(), Zod), dan dokumentasi endpoint.

2.5 Pemrograman Berorientasi Objek

OOP didasarkan pada empat pilar: encapsulation, inheritance, polymorphism, dan abstraction. Pada Laravel, model Eloquent meng-extend kelas Model, controller meng-extend Controller, dan relasi morphMany merupakan bentuk polimorfisme yang nyata pada level domain.

2.6 Reuse via Library / Komponen Pre-Existing

Menulis ulang fungsionalitas yang sudah tersedia di ekosistem adalah anti-pattern. Composer (PHP) dan npm (JS) memungkinkan kita mengkurasi library berkualitas: Spatie Permission untuk RBAC, Midtrans SDK untuk pembayaran, shadcn/ui & Radix UI untuk komponen UI yang accessible, dsb.

2.7 Akses Basis Data & ORM

Akses basis data modern menghindari query mentah demi keamanan (anti SQL injection) dan keterbacaan. Eloquent ORM Laravel menyediakan fluent interface, eager loading untuk menghindari N+1, dan migrations sebagai versi schema yang dapat diaudit.

2.8 Pengujian Unit (Unit & Feature Test)

Unit test memverifikasi perilaku komponen kecil secara terisolasi; feature test memverifikasi interaksi lintas komponen (HTTP request → response). Pest PHP adalah framework testing ekspresif yang berjalan di atas PHPUnit. SQLite in-memory dipakai sebagai database test agar cepat dan reproducible.

4.1.1 Struktur Folder Mengikuti Konvensi Laravel

order-store/
├── app/
│   ├── Http/Controllers/        ← Controller (ProductController, ShipmentController, ...)
│   ├── Http/Middleware/         ← Middleware kustom
│   ├── Http/Requests/           ← Form Request validation

Penjelasan: struktur folder mengikuti convention Laravel: Controller, Model, dan Middleware berada pada subfolder yang sudah diharapkan. Folder app/Services/ ditambahkan untuk menampung business logic kompleks (mis. perhitungan ongkir) supaya Controller tetap ringkas. Frontend React/Inertia hidup di resources/js/ sehingga backend (PHP) dan frontend (TSX) terpisah tetapi tetap dalam satu monorepo.

4.1.2 PSR-4 Autoloading & Namespace

// composer.json
"autoload": {
    "psr-4": {
        "App\\": "app/",
        "Database\\Factories\\": "database/factories/",
        "Database\\Seeders\\": "database/seeders/"
    }
}

Penjelasan: blok psr-4 memetakan namespace ke folder. Artinya kelas App\Models\Product otomatis dimuat dari app/Models/Product.php tanpa perlu require manual. Hal ini menjamin tidak ada nama kelas yang bentrok dan membuat code search antar editor menjadi sangat cepat.

4.1.3 Single Responsibility & Separation of Concerns

Controller bertanggung jawab atas alur HTTP, Model atas struktur data, dan Service Layer atas logika bisnis non-trivial (mis. DeliveryService menghitung ongkir).

4.1.4 Konsistensi Gaya via Tooling

Laravel Pint, Prettier, ESLint, dan TypeScript --noEmit dijalankan agar gaya penulisan seragam tanpa beban manual.

Hasil Penerapan

Onboarding pengembang baru menjadi cepat karena struktur project bersifat self-explanatory, dan code review dapat fokus pada substansi bukan format.

4.2 Unit 2Melakukan Debugging

4.2.1 Logging Terstruktur via Facade Log

Kegagalan integrasi pihak ketiga ditangkap dan dicatat ke log tanpa menghentikan alur utama, sehingga developer tetap punya jejak audit ketika terjadi kegagalan diam-diam (silent failure).

// app/Services/DeliveryService.php
use Illuminate\Support\Facades\Log;

try {
    $response = Http::withHeaders([
        'User-Agent' => config('app.name', 'Laravel').' delivery geocoder',
    ])->timeout(8)->get(config('delivery.geocode_url'), [
        'q' => $query,
        'format' => 'json',
        'limit' => 1,
        'countrycodes' => config('delivery.geocode_country'),
    ]);

    // ... proses response
} catch (\Throwable $e) {
    Log::warning('Delivery geocoding failed: '.$e->getMessage());
}

Penjelasan: panggilan ke API OpenStreetMap Nominatim dibungkus dengan try/catch. Apabila terjadi exception (timeout, DNS gagal, format JSON tidak valid), pesan kesalahan dicatat pada channel log Laravel via Log::warning(). Cara ini lebih baik daripada die() atau membiarkan exception bocor — pengguna tetap mendapat respons, sementara developer punya jejak audit untuk investigasi melalui storage/logs/laravel.log.

4.2.2 Defensive Fallback agar Sistem Tetap Berjalan

Saat geocoding gagal, sistem tidak melemparkan error ke pengguna; pengguna tetap memperoleh estimasi ongkir dari fallback_fee, sementara developer memperoleh log untuk investigasi.

if (! $coords) {
    return [
        'distance_km'   => null,
        'shipping_cost' => (int) config('delivery.fallback_fee'),
        'geocoded'      => false,
    ];
}

Penjelasan: blok if (! $coords) menangani kasus ketika geocoding tidak berhasil menemukan koordinat. Daripada throw exception, sistem mengembalikan tarif tetap dari konfigurasi (delivery.fallback_fee) dan menandai geocoded => false. Frontend kemudian dapat menampilkan pesan informatif (mis. “estimasi sementara”) — proses checkout tetap berlanjut dan tidak ada user-facing error.

4.2.3 Real-Time Log Tailing dengan Laravel Pail

Paket laravel/pail diaktifkan sebagai require-dev dan dapat dipanggil dengan:

php artisan pail            # streaming semua log
php artisan pail --filter=ERROR  # filter level error saja

Penjelasan: perintah pail men-stream log Laravel ke terminal secara real-time. Sangat berguna saat reproduksi bug — developer dapat melihat detail event (mis. query SQL, exception, deprecation) tepat saat aksi dilakukan di browser. Argumen --filter memungkinkan menyaring berdasarkan level, sehingga tidak terkubur oleh log informatif.

4.2.4 Exception Penanganan Aman pada File Storage

Saat menghapus gambar produk yang sudah tidak terpakai, kegagalan menghapus file fisik dibungkus try/catch agar transaksi DB tidak gagal hanya karena file sudah tidak ada di disk.

// app/Http/Controllers/ProductController.php
foreach ($toDelete as $img) {
    try {
        Storage::disk('public')->delete($img->image_path);
    } catch (\Throwable $e) {
        // optional: log — file sudah tidak ada, abaikan
    }
    $img->delete();
}

Penjelasan: setiap gambar yang akan dihapus diproses dalam dua langkah — hapus file fisik di disk, kemudian hapus baris pada tabel images. Kegagalan menghapus file (mis. file sudah terlanjur dihapus admin lain) tidak boleh menghentikan proses. Penangkapan \Throwable menjamin kondisi tersebut tidak merusak transaksi besar yang sedang berlangsung.

4.2.5 Validasi Sebagai Bentuk Pencegahan Bug

Sebagian besar bug pada aplikasi web berasal dari input yang tidak terduga. Validasi pada controller (lihat 4.4) berperan sebagai defensive line pertama.

4.3.1 PHPDoc dengan Tipe Return Terstruktur

4.3.2 Dokumentasi Konfigurasi yang Self-Explanatory

Setiap parameter konfigurasi pada config/delivery.php dilengkapi komentar blok yang menjelaskan mengapa nilainya ada dan kapan harus diubah.

// config/delivery.php
return [
    /*
    |--------------------------------------------------------------------------
    | Delivery Origin
    |--------------------------------------------------------------------------
    |
    | The coordinates the package is dispatched from (your store / warehouse).
    | Distance to the customer is measured from this point. Defaults to a
    | central Jakarta coordinate; override per deployment via .env.
    |
    */
    'origin' => [
        'lat' => (float) env('DELIVERY_ORIGIN_LAT', -6.200000),
        'lng' => (float) env('DELIVERY_ORIGIN_LNG', 106.816666),
    ],

    /*
    |--------------------------------------------------------------------------
    | Pricing
    |--------------------------------------------------------------------------
    |
    | Flat rate charged per kilometre (rounded up, minimum 1 km). When the
    | address cannot be geocoded we fall back to a flat fee so checkout never
    | breaks.
    |
    */
    'rate_per_km'  => (int) env('DELIVERY_RATE_PER_KM', 2000),
    'fallback_fee' => (int) env('DELIVERY_FALLBACK_FEE', 15000),
];

Penjelasan: setiap kunci konfigurasi diberi komentar blok bergaya Laravel. Komentar menjelaskan tujuan nilai (mis. titik origin pengiriman), default, dan cara override via file .env. Tipe nilai juga di-cast eksplisit (mis. (float), (int)) agar konfigurasi selalu konsisten meskipun ENV dibaca sebagai string.

4.3.3 Komentar Inline Pada Logika yang Tidak Obvious

// Try the most specific query first, then progressively coarser ones.
// A vague house-level address often isn't in OpenStreetMap, so falling
// back to the official village (kelurahan/desa) then district then
// city/province still gives a usable distance instead of the flat fee.
$candidates = array_unique(array_filter([
    implode(', ', array_filter([$address, $village, $district, $city, $province, $postalCode])),
    implode(', ', array_filter([$village, $district, $city, $province])),
    // ...
]));

Penjelasan: komentar menjelaskan mengapa ada beberapa kandidat query, bukan sekadar apa. Tanpa konteks ini, pembaca akan kebingungan kenapa alamat dipecah menjadi varian yang semakin kasar. Inilah komentar “berhak hidup”: yang menjelaskan keputusan desain, bukan parafrase kode.

4.3.4 Komentar Pengelompokan Rute

Pada routes/web.php setiap kelompok izin diberi penanda agar mudah ditelusuri ketika menambah rute baru:

// 1. Product Create
Route::middleware(['permission:product.create'])->group(function () { /* ... */ });

// 2. Edit/Update Permissions
Route::middleware(['permission:product.edit'])->group(function () { /* ... */ });

// 3. View Product
Route::middleware(['permission:product.view'])->group(function () { /* ... */ });

// 4. Delete Product
Route::middleware(['permission:product.delete'])->group(function () { /* ... */ });

Penjelasan: rute CRUD produk dibagi ke 4 kelompok berdasarkan izin (create, edit, view, delete). Komentar bernomor membuat skema CRUD mudah ditemukan dengan Ctrl+F. Ketika menambah rute baru, developer cukup memasukkannya pada grup izin yang tepat.

4.4 Unit 4Menggunakan Spesifikasi Program

4.4.1 Spesifikasi via PHP Type Hints & Return Types

Setiap parameter dan nilai kembalian fungsi diberi tipe yang ketat sehingga kontrak fungsi terbaca dari signaturnya:

public function quote(
    string $address,
    ?string $city = null,
    ?string $province = null,
    ?string $postalCode = null,
    ?string $district = null,
    ?string $village = null
): array { /* ... */ }

protected function geocode(string $query): ?array { /* ... */ }
protected function haversine(float $lat1, float $lon1, float $lat2, float $lon2): float { /* ... */ }

public function store(Request $request): RedirectResponse { /* ... */ }

Penjelasan: tanda ? pada ?string berarti parameter boleh null. Tipe pengembalian (: array, : ?array, : float, : RedirectResponse) menjadi spesifikasi yang dipaksakan PHP saat runtime — bila implementasi mengembalikan tipe lain, error langsung muncul.

4.4.2 Spesifikasi Validasi Input

Validasi Laravel adalah bentuk spesifikasi yang dijalankan saat runtime untuk memastikan kontrak input dari klien terpenuhi:

// app/Http/Controllers/ProductController.php
$validated = $request->validate([
    'title'        => ['required', 'string', 'max:255'],
    'description'  => ['required', 'string'],
    'price'        => ['required', 'numeric'],
    'is_featured'  => ['required'],
    'is_limited'   => ['required'],
    'stock'        => ['required', 'integer'],
    'category_id'  => ['required', 'exists:categories,id'],
    'images'       => ['sometimes', 'array'],
    'images.*'     => ['image', 'mimes:jpg,jpeg,png,webp', 'max:4096'],
]);

Penjelasan: aturan didefinisikan sebagai array. required memaksa field harus ada; exists:categories,id memastikan category_id benar-benar ada di tabel categories (sekaligus pencegahan foreign key violation); images.* dengan rule image|mimes|max:4096 menjamin setiap file adalah gambar (jpg/jpeg/png/webp) berukuran maksimal 4 MB. Bila gagal, Laravel otomatis mengembalikan response 422 dengan pesan kesalahan terstruktur.

4.4.3 Spesifikasi Tipe di Frontend (TypeScript)

Komponen halaman menerima props yang tipenya didefinisikan di resources/js/types/:

// resources/js/pages/dashboard.tsx
import { dashboard } from '@/routes';
import { type BreadcrumbItem } from '@/types';

const breadcrumbs: BreadcrumbItem[] = [
    { title: 'Dashboard', href: dashboard().url },
];

export default function Dashboard() {
    return (
        <AppLayout breadcrumbs={breadcrumbs}>
            <Head title="Dashboard" />
            <SectionCards />
            <ChartAreaInteractive />
        </AppLayout>
    );
}

Penjelasan: tipe BreadcrumbItem[] diimpor sebagai type-only import sehingga tidak masuk bundle JavaScript. Bila ada properti title atau href yang lupa diset atau salah ketik, TypeScript akan menggagalkan compile sebelum kode dijalankan di browser.

4.4.4 Spesifikasi via Laravel Wayfinder

Wayfinder meng-generate helper rute bertipe sehingga kesalahan nama rute terdeteksi pada saat compile-time, bukan runtime. Contoh: dashboard().url akan gagal kompilasi bila rute tidak terdaftar.

4.4.5 Spesifikasi Skema Database via Migrations

Spesifikasi struktur tabel diekspresikan secara deklaratif dalam migration, lengkap dengan foreign key constraint:

// database/migrations/2025_10_02_114918_create_products_table.php
Schema::create('products', function (Blueprint $table) {
    $table->id();
    $table->string('title');
    $table->text('description');
    $table->bigInteger('price');
    $table->integer('stock')->default(0);
    $table->unsignedBigInteger('category_id')->default(0);
    $table->foreign('category_id')->references('id')->on('categories')->onDelete('cascade');
    $table->timestamps();
});

Penjelasan: blok ini mendefinisikan tabel products secara deklaratif. Kolom price menggunakan bigInteger agar mampu menyimpan harga dalam Rupiah tanpa kehilangan presisi (tidak ada floating-point). onDelete('cascade') memastikan ketika kategori dihapus, produk turunannya ikut dihapus — sehingga tidak ada data orphan.

4.5 Unit 5Mengimplementasikan Pemrograman Berorientasi Objek

4.5.1 Inheritance — Model Extends Eloquent Model

// app/Models/Product.php
namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;

class Product extends Model
{
    use HasFactory;

    protected $fillable = [
        'title', 'description', 'is_featured', 'is_limited',
        'stock', 'category_id', 'price',
    ];

    protected $casts = [
        'options' => 'array',
    ];

    public function images()
    {
        return $this->morphMany(Image::class, 'imageable');
    }

    public function category()
    {
        return $this->belongsTo(Category::class);
    }
}

Penjelasan: kelas Product mewarisi Illuminate\Database\Eloquent\Model, sehingga otomatis mendapat kemampuan CRUD (create, find, update, delete) tanpa menulis SQL. Properti $fillable adalah mass-assignment guard — hanya kolom dalam daftar yang boleh diisi via array, mencegah eksploitasi yang menyusupkan kolom sensitif. $casts mengonversi kolom JSON menjadi array PHP secara otomatis.

4.5.2 Polymorphism — Relasi morphMany

Tabel images dapat menjadi gambar untuk banyak entitas berbeda (Product, User, Category) tanpa duplikasi tabel. Inilah bentuk polimorfisme pada level domain.

4.5.3 Encapsulation — Private Method & Visibility Modifier

Method utilitas internal pada controller di-private-kan agar tidak diakses dari luar:

// app/Http/Controllers/ProductController.php
private function normalizeImagePath(?string $input): ?string
{
    if (!$input) return null;
    $input = trim($input);
    $input = preg_replace('#^https?://[^/]+/storage/#i', '', $input);
    $input = preg_replace('#^/storage/#i', '', $input);
    if (!preg_match('#^uploads/products/#i', $input)) {
        return null;
    }
    return $input;
}

private function mapProductData($product) { /* ... */ }

Penjelasan: method normalizeImagePath dibuat private karena hanya bermanfaat di dalam ProductController. Modifier akses ini adalah bentuk enkapsulasi: detail implementasi (mis. regex pembersih URL) tersembunyi dan tidak dapat dipanggil oleh kode di luar kelas. Method juga melakukan path traversal guard dengan menolak path di luar uploads/products/.

// DeliveryService — geocode & haversine adalah detail internal
protected function geocode(string $query): ?array { /* ... */ }
protected function haversine(/* ... */): float { /* ... */ }

Penjelasan: dua method ini dibuat protected — boleh dipanggil oleh subkelas (misalnya jika nanti ada FastDeliveryService extends DeliveryService), tetapi tidak boleh dipanggil langsung dari luar. Ini menjaga kontrak publik tetap kecil (hanya quote()).

4.5.4 Abstraction — Service Layer

Controller tidak mengetahui detail geocoding dan kalkulasi jarak. Ia hanya memanggil $service->quote(...) dan menerima hasil:

$service = app(DeliveryService::class);
$quote = $service->quote($address, $city, $province, $postalCode, $district, $village);
// $quote sekarang berisi distance_km, shipping_cost, geocoded

Penjelasan: app(DeliveryService::class) meminta instance dari Laravel Service Container. Controller hanya peduli pada kontrak publik (quote()) — apakah di belakangnya pakai OpenStreetMap, Google Maps, atau hardcode tidak menjadi urusan controller. Inilah abstraction: detail rumit disembunyikan di balik antarmuka sederhana.

4.5.5 Trait — Pewarisan Horizontal

HasFactory di-use pada model untuk menambahkan kapabilitas factory tanpa perlu pewarisan vertikal. Ini contoh composition over inheritance.

4.5.6 OOP di Frontend — Komponen sebagai Kelas Logis

Komponen React dengan TypeScript dapat dianggap sebagai “objek” yang memiliki props (kontrak input) dan state (enkapsulasi internal). Pemisahan ke berkas seperti app-header.tsx, app-sidebar.tsx, dan nav-main.tsx mencerminkan single responsibility di sisi UI.

4.6.4 Layanan Pihak Ketiga melalui HTTP Client Bawaan

OpenStreetMap Nominatim diakses lewat Laravel HTTP client (Illuminate\Support\Facades\Http) — fasilitas pre-existing yang sudah menangani timeout, headers, dan parsing JSON.

$response = Http::withHeaders([
    'User-Agent' => config('app.name', 'Laravel').' delivery geocoder',
])->timeout(8)->get(config('delivery.geocode_url'), [
    'q' => $query,

Penjelasan: Http::withHeaders()->timeout(8)->get() adalah fluent interface bawaan Laravel. 4.7 Unit 7Menerapkan Akses Basis Data

4.7.1 Migrations sebagai Versi Schema

Setiap perubahan struktur tabel diekspresikan sebagai berkas migration sehingga seluruh tim memiliki schema yang konsisten dan dapat dikembalikan (rollback):

database/migrations/
├── 0001_01_01_000000_create_users_table.php
├── 2025_10_02_114913_create_categories_table.php
├── 2025_10_02_114918_create_products_table.php
├── 2025_10_05_141521_images.php
├── 2026_01_13_084012_create_permission_tables.php
├── 2026_05_12_025849_create_transactions_table.php
├── 2026_06_04_014251_create_shipments_table.php
└── 2026_06_04_060000_add_shipping_village_to_transactions_table.php

Penjelasan: nama file diawali timestamp (YYYY_MM_DD_HHMMSS) agar urutan eksekusi migration deterministik. Perubahan kecil (mis. menambah kolom shipping_village) dibuat sebagai migration terpisah, bukan mengubah migration lama — sehingga sejarah perubahan schema tetap utuh dan dapat di-rollback.

4.7.2 Foreign Key & Cascade

Schema::create('products', function (Blueprint $table) {
    $table->id();
    $table->string('title');
    $table->text('description');
    $table->bigInteger('price');
    $table->integer('stock')->default(0);
    $table->unsignedBigInteger('category_id')->default(0);
    $table->foreign('category_id')->references('id')->on('categories')->onDelete('cascade');
    $table->timestamps();
});

Penjelasan: integritas referensial dijamin oleh foreign key constraint. Database akan menolak category_id yang tidak ada. Klausa onDelete('cascade') memastikan tidak ada produk orphan ketika kategorinya dihapus.

4.7.3 Eloquent ORM — Anti SQL Injection

Setiap query menggunakan Eloquent atau Query Builder dengan parameter binding, sehingga input pengguna tidak pernah di-concatenate ke string SQL.

// Filter produk dinamis pada ProductController::getFilteredProducts
$query = Product::with('images');

if ($request->filled('search')) {
    $search = $request->input('search');
    $query->where(function ($q) use ($search) {
        $q->where('title', 'like', "%{$search}%")
          ->orWhereHas('category', function ($cq) use ($search) {
              $cq->where('name', 'like', "%{$search}%");
          });
    });
}

if ($request->filled('category_id')) {
    $query->where('category_id', $request->category_id);
}

if ($request->filled('min_price')) {
    $query->where('price', '>=', $request->min_price);
}

$paginator = $query->paginate($perPage, ['*'], 'page', $page);

Penjelasan: filter dibangun secara incremental menggunakan conditional clauses. Setiap nilai yang berasal dari pengguna ($search, category_id, min_price) ditangani oleh Query Builder yang melakukan parameter binding otomatis — jadi aman dari SQL Injection meskipun secara sintaks terlihat seperti string interpolation. orWhereHas melakukan subquery pada relasi category, dan paginate() menghasilkan response dengan metadata pagination.

4.7.4 Eager Loading — Mencegah N+1

// Memuat relasi images bersamaan agar tidak terjadi N+1 query
$product = Product::with('images')->findOrFail($id);

// Multi-relasi
$product = Product::with(['images', 'category'])->find($id);

Penjelasan: tanpa with(), Laravel akan menjalankan query terpisah untuk setiap relasi tiap kali diakses — masalah klasik N+1. Dengan eager loading, semua data dimuat dalam satu set query efisien: 1 untuk produk, 1 untuk seluruh image, 1 untuk seluruh category. findOrFail memunculkan response 404 otomatis bila ID tidak ditemukan.

4.7.5 Atomicity dengan Database Transaction

DB::transaction(function () use ($request, $validated) {
    $product = Product::create([
        'title'        => $validated['title'],
        'description'  => $validated['description'],
        'price'        => $validated['price'],
        'stock'        => $validated['stock'],
        'is_limited'   => $validated['is_limited'] === 'true' ? 1 : 0,
        'is_featured'  => $validated['is_featured'] === 'true' ? 1 : 0,
        'category_id'  => $validated['category_id'],
    ]);

    if ($request->hasFile('images')) {
        foreach ($request->file('images') as $file) {
            $path = $file->store('uploads/products', 'public');
            $product->images()->create([
                'image_path' => $path,
                'is_primary' => 1,
            ]);
        }
    }
});

Penjelasan: seluruh operasi (membuat produk + menyimpan banyak gambar) dibungkus DB::transaction(). Jika satu langkah gagal — misal salah satu file gagal disimpan ke disk — semua perubahan database otomatis di-rollback. Tidak akan terjadi kondisi produk “setengah jadi” (ada di DB tapi tanpa gambar). Inilah atomicity: all-or-nothing.

4.7.6 Casting Tipe Kolom JSON / Date

// app/Models/Transaction.php
protected $casts = [
    'items' => 'array',
    'paid_at' => 'datetime',
];

Penjelasan: kolom items disimpan di database sebagai string JSON, namun ketika diakses di PHP otomatis menjadi array berkat cast. paid_at menjadi instance Carbon sehingga developer dapat memanggil $tx->paid_at->diffForHumans() tanpa konversi manual. Ini menghilangkan boilerplate di tiap pemanggilan.

4.8 Unit 8Melaksanakan Pengujian Unit Program

4.8.1 Konfigurasi PHPUnit / Pest

Konfigurasi phpunit.xml mendefinisikan dua suite: Unit (logika murni) dan Feature (HTTP request → response). Database test menggunakan SQLite in-memory agar cepat dan reproducible.

<phpunit ...>
    <testsuites>
        <testsuite name="Unit">
            <directory>tests/Unit</directory>
        </testsuite>
        <testsuite name="Feature">
            <directory>tests/Feature</directory>
        </testsuite>
    </testsuites>
    <php>
        <env name="APP_ENV" value="testing"/>
        <env name="DB_CONNECTION" value="sqlite"/>
        <env name="DB_DATABASE" value=":memory:"/>
        <env name="CACHE_STORE" value="array"/>
        <env name="MAIL_MAILER" value="array"/>
        <env name="QUEUE_CONNECTION" value="sync"/>
    </php>
</phpunit>

Penjelasan: DB_DATABASE=":memory:" menjalankan database SQLite di RAM — sangat cepat dan otomatis hilang setelah test selesai, sehingga test tidak meninggalkan sampah. CACHE_STORE=array dan QUEUE_CONNECTION=sync membuat cache dan queue jalan in-memory tanpa side effect. Email juga di-capture di array, bukan dikirim sungguhan.

4.8.2 Bootstrap Pest dengan RefreshDatabase

// tests/Pest.php
pest()->extend(Tests\TestCase::class)
    ->use(Illuminate\Foundation\Testing\RefreshDatabase::class)
    ->in('Feature');

expect()->extend('toBeOne', function () {
    return $this->toBe(1);
});

Penjelasan: trait RefreshDatabase menjalankan migrate:fresh sebelum setiap test sehingga state database selalu bersih dan deterministik. expect()->extend('toBeOne', ...) menambahkan custom matcher agar test dapat ditulis dengan lebih ekspresif: expect($count)->toBeOne().

4.8.3 Smoke Test Halaman Utama

// tests/Feature/ExampleTest.php
it('returns a successful response', function () {
    $response = $this->get('/');

    $response->assertStatus(200);
});

Penjelasan: test paling dasar — “halaman home masih nyala”. Pest menggunakan function-style it('...') yang dibaca seperti kalimat. Bila dependency penting (misalnya provider) rusak, test ini gagal duluan sebelum test lain berjalan.

4.8.4 Feature Test — Otorisasi Dashboard

Memastikan pengguna guest dialihkan ke halaman login dan pengguna terautentikasi dapat mengakses dashboard:

// tests/Feature/DashboardTest.php
use App\Models\User;

test('guests are redirected to the login page', function () {
    $this->get(route('dashboard'))->assertRedirect(route('login'));
});

test('authenticated users can visit the dashboard', function () {
    $this->actingAs($user = User::factory()->create());

    $this->get(route('dashboard'))->assertOk();
});

Penjelasan: test pertama memverifikasi middleware auth bekerja — pengguna tak login dialihkan ke /login. Test kedua memakai User::factory() untuk membuat pengguna palsu, lalu actingAs() men-simulasi login. Setelah login, akses ke /dashboard harus mengembalikan 200 (OK). Dua test ini melindungi alur autentikasi dari regresi.

4.8.5 Feature Test — Modul Autentikasi & Pengaturan

Folder tests/Feature/Auth/ dan tests/Feature/Settings/ berisi test untuk:

  • Login, registrasi, password reset, dan verifikasi email.
  • Pengaturan profil, kata sandi, dan 2FA.

4.8.6 Menjalankan Test

# Menjalankan seluruh suite
php artisan test

# Suite tertentu
php artisan test --testsuite=Feature
php artisan test --testsuite=Unit

# File spesifik
php artisan test tests/Feature/DashboardTest.php

# Mode paralel
php artisan test --parallel

Penjelasan: perintah php artisan test adalah wrapper Laravel di atas Pest/PHPUnit dengan output yang lebih ramah dan ikon yang mudah dibaca. Flag --parallel menjalankan test secara paralel di beberapa proses CPU, sangat berguna ketika jumlah test sudah besar.

Manfaat

Pengujian otomatis menjadi safety net ketika kode di-refactor. Setiap perubahan dapat diverifikasi dalam hitungan detik, bukan menit; regresi pada alur kritis (autentikasi, dashboard, dsb.) tertangkap sebelum sampai ke pengguna.

BAB V — KESIMPULAN

Berdasarkan implementasi yang telah didokumentasikan pada BAB IV, proyek Order Store telah menerapkan delapan unit kompetensi pemrograman secara terpadu:

  1. Unit 1 — Best Practice: struktur folder Laravel, PSR-4, SOLID, dan tooling otomatis (Pint, Prettier, ESLint, TypeScript).
  2. Unit 2 — Debugging: Log facade, defensive fallback, try/catch pada operasi I/O, dan Laravel Pail untuk real-time tailing.
  3. Unit 3 — Dokumen Kode: PHPDoc dengan tipe array terstruktur, komentar konfigurasi yang menjelaskan mengapa, dan penanda bagian rute pada routes/web.php.
  4. Unit 4 — Spesifikasi Program: type hints PHP 8, TypeScript, validasi $request->validate(), dan migrations sebagai spesifikasi schema.
  5. Unit 5 — OOP: inheritance (Model, Controller), polimorfisme via morphMany, enkapsulasi via private/protected, dan abstraksi melalui Service Layer.
  6. Unit 6 — Library Pre-Existing: Spatie Permission, Fortify, Midtrans SDK, Inertia, Radix UI, shadcn/ui, TanStack Table, Recharts, dsb.
  7. Unit 7 — Akses Basis Data: Migrations, foreign key cascade, Eloquent dengan parameter binding, eager loading, transaksi atomik, dan casting.
  8. Unit 8 — Pengujian Unit Program: Pest PHP dengan SQLite in-memory, dua suite (Unit + Feature), serta test untuk autentikasi, dashboard, dan pengaturan.

Penerapan kedelapan unit ini tidak berdiri sendiri; mereka saling mendukung. Misalnya, type hints (Unit 4) memperkuat debugging (Unit 2) karena banyak kelas bug ditolak compile-time; OOP yang baik (Unit 5) membuat unit test (Unit 8) menjadi mudah ditulis; dan dokumentasi (Unit 3) memudahkan penerapan best practice (Unit 1) oleh anggota tim baru.

Rekomendasi Pengembangan Lanjutan

  • Memindahkan validasi besar dari controller ke Form Request kelas tersendiri untuk meningkatkan keterujian.
  • Menambahkan Policy Laravel untuk otorisasi berbasis instance (mis. courier X hanya boleh mengubah shipment-nya sendiri).
  • Menulis unit test langsung untuk DeliveryService dengan HTTP fake agar logika ongkir terjamin tanpa memanggil API Nominatim.
  • Menambah cakupan feature test alur checkout, webhook Midtrans, dan shipping quote.
  • Menerapkan commit hook (Husky/lefthook) agar Pint, Prettier, ESLint, TypeScript, dan Pest dijalankan otomatis pra-commit.
  • Menambah dokumentasi API contract (OpenAPI) untuk endpoint publik (get-all-products, get-filtered-products, dsb.).
