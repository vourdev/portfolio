BAB I — PENDAHULUAN

1.1 Latar Belakang

Sektor e-commerce di Indonesia mengalami pertumbuhan signifikan dalam beberapa tahun terakhir, didukung oleh kemudahan akses internet dan perubahan perilaku konsumen yang lebih memilih berbelanja secara daring. Dalam konteks pengembangan aplikasi e-commerce seperti Order Store, kompleksitas sistem meningkat seiring bertambahnya fitur, seperti manajemen produk, kategori, transaksi pembayaran (Midtrans), pengiriman (kurir & geocoding), serta kontrol hak akses (RBAC).

Tanpa disiplin penulisan kode yang baik, kompleksitas tersebut dapat dengan cepat berubah menjadi technical debt: kode menjadi sulit dibaca, sulit dirawat, rawan bug, serta sulit untuk ditambahkan fitur baru. Oleh karena itu, mengikuti guidelines dan best practice dalam menulis kode bukanlah pilihan, melainkan kebutuhan agar sistem dapat tumbuh secara berkelanjutan (scalable) dan aman.

Portfolio ini mendokumentasikan bagaimana proyek Order Store menerapkan kaidah-kaidah penulisan kode profesional berdasarkan standar komunitas Laravel, PHP-FIG (PSR), React, serta praktik keamanan OWASP.

1.2 Tujuan Penyusunan Portfolio

  1. Mendokumentasikan prinsip-prinsip penulisan kode yang diterapkan pada proyek Order Store.
  2. Memberikan contoh nyata implementasi best practice langsung dari kode sumber proyek.
  3. Menjadi rujukan internal tim agar gaya penulisan kode tetap konsisten ke depannya.
  4. Menjadi bukti kompetensi pengembangan aplikasi web profesional.

1.3 Ruang Lingkup

  • Codebase: project order-store dengan stack Laravel 12, Inertia.js 2, React 19, TypeScript 5, dan Tailwind CSS 4.
  • Fokus: kaidah penulisan kode (clean code, SOLID, DRY, KISS), arsitektur MVC, keamanan, serta tooling kualitas kode.
  • Di luar lingkup: dokumentasi fungsional/UAT, infrastruktur deployment, dan analisis performa load testing.

BAB II — LANDASAN TEORI

2.1 Clean Code dan Coding Standard

Robert C. Martin (2008) dalam Clean Code menjelaskan bahwa kode bersih adalah kode yang mudah dibaca oleh manusia, bukan hanya oleh mesin. Indikatornya meliputi: penamaan yang ekspresif, fungsi yang pendek dan melakukan satu hal, tidak ada duplikasi, serta dapat dipahami tanpa banyak komentar. Coding standard (gaya penulisan) memastikan setiap anggota tim menulis kode dengan format yang seragam sehingga code review menjadi lebih cepat dan diff lebih informatif.

2.2 Prinsip SOLID

  • Single Responsibility Principle — sebuah kelas/fungsi hanya memiliki satu alasan untuk berubah.
  • Open/Closed Principle — entitas terbuka untuk ekstensi, tertutup untuk modifikasi.
  • Liskov Substitution Principle — subkelas harus dapat menggantikan kelas induknya tanpa merusak perilaku.
  • Interface Segregation Principle — banyak interface kecil lebih baik daripada satu interface besar.
  • Dependency Inversion Principle — bergantung pada abstraksi, bukan implementasi.

2.3 Don’t Repeat Yourself (DRY) & KISS

DRY (Hunt & Thomas, 1999) menekankan bahwa setiap pengetahuan dalam sistem harus memiliki satu representasi otoritatif. KISS (Keep It Simple, Stupid) mendorong solusi paling sederhana yang menyelesaikan masalah, menghindari abstraksi prematur.

2.4 Standar PSR-12 dan Laravel Coding Style

PHP-FIG menerbitkan PSR-12 sebagai standar gaya penulisan kode PHP modern (indentasi 4 spasi, deklarasi namespace, urutan use statement, dsb.). Laravel memiliki gaya bawaan yang merupakan ekstensi PSR-12 dan dapat ditegakkan otomatis dengan tool laravel/pint.

2.5 Separation of Concerns & MVC

Pola Model-View-Controller (MVC) memisahkan logika data (Model), presentasi (View), dan kontrol alur (Controller). Pada arsitektur Inertia.js, peran View diisi oleh komponen React yang menerima props langsung dari Controller Laravel — menjaga frontend dan backend tetap terpisah tetapi terintegrasi rapi.

2.6 Security Best Practice (OWASP)

OWASP Top 10 mencantumkan ancaman seperti Broken Access Control, Injection, Cross-Site Scripting, dan Insecure Design. Aplikasi yang baik harus memitigasi ancaman ini melalui validasi input, parameter binding pada query, otorisasi terstruktur, serta proteksi CSRF.

BAB IV — IMPLEMENTASI BEST PRACTICE

4.1 Struktur Folder Sesuai Konvensi Laravel

Project mengikuti struktur direktori standar Laravel sehingga setiap pengembang baru dapat segera memahami letak setiap bagian sistem tanpa perlu dokumentasi tambahan:

order-store/
├── app/
│   ├── Http/Controllers/        ← Controller (ProductController, ShipmentController, ...)
│   ├── Http/Middleware/         ← Middleware kustom
│   ├── Http/Requests/           ← Form Request validation
│   ├── Models/                  ← Eloquent models (Product, Category, Image, Transaction, Shipment, User)
│   ├── Providers/               ← Service providers (AppServiceProvider, FortifyServiceProvider)
│   └── Services/                ← Service layer (DeliveryService)
├── database/migrations/         ← Versi schema database
├── resources/js/                ← Source TSX (React + Inertia)
│   ├── pages/                   ← Halaman (admin, auth, dashboard, users, settings)
│   ├── components/              ← Reusable components (shadcn-style)
│   ├── layouts/                 ← Layout wrapper
│   ├── hooks/, lib/, types/     ← Utilities, custom hooks, type definitions
│   └── routes/, wayfinder/      ← Type-safe route helpers (Wayfinder generated)
├── routes/                      ← web.php, auth.php, settings.php, console.php
└── config/, public/, tests/

Prinsip yang diterapkan

Convention over Configuration. Dengan mengikuti konvensi bawaan Laravel, kami menghindari pengulangan keputusan struktur. Anggota tim baru dapat onboarding lebih cepat.

4.2 PSR-4 Autoloading & Namespace

Setiap kelas PHP dipetakan dengan namespace sesuai PSR-4 melalui composer.json:

"autoload": {
    "psr-4": {
        "App\\": "app/",
        "Database\\Factories\\": "database/factories/",
        "Database\\Seeders\\": "database/seeders/"
    }
}

Penggunaan namespace memungkinkan setiap kelas memiliki identitas unik dan dapat di-autoload tanpa require manual. Contoh penerapan dalam controller:

namespace App\Http\Controllers;

use App\Models\Category;
use App\Models\Product;
use Illuminate\Http\RedirectResponse;
use Illuminate\Http\Request;
use Inertia\Inertia;

Urutan use dikelola secara alfabetis dan dipisahkan per vendor — sebuah praktik yang ditegakkan otomatis oleh Laravel Pint.

4.3 MVC dan Separation of Concerns

Setiap permintaan HTTP dilayani oleh tiga lapisan terpisah:

  • Model: app/Models/Product.php — mendefinisikan tabel, kolom fillable, casting, dan relasi.
  • Controller: ProductController — mengelola alur permintaan, validasi, dan response.
  • View: komponen React via Inertia, mis. resources/js/pages/admin/products/index.tsx.
// app/Http/Controllers/ProductController.php
public function index()
{
    return Inertia::render('admin/products/index', [

4.4 Eloquent ORM & Relasi Polymorphic

Sebagai pengganti query SQL mentah, kami menggunakan Eloquent ORM yang aman dari SQL injection (parameter binding otomatis) dan ekspresif:

// app/Models/Product.php
class Product extends Model
{
    use HasFactory;

    protected $fillable = [
        'title', 'description', 'is_featured', 'is_limited',
        'stock', 'category_id', 'price',
    ];

    public function images()
    {
        return $this->morphMany(Image::class, 'imageable');
    }

    public function category()
    {
        return $this->belongsTo(Category::class);
    }
}

Relasi morphMany memungkinkan satu tabel images dipakai untuk berbagai entitas (Product, User, dll.) tanpa duplikasi tabel.

4.5 Validasi Input pada Boundary

Sesuai prinsip Trust internal code, validate at system boundaries, setiap input dari klien divalidasi langsung pada controller sebelum diproses lebih lanjut:

// app/Http/Controllers/ProductController.php
public function store(Request $request): RedirectResponse
{
    $validated = $request->validate([
        'title'        => ['required', 'string', 'max:255'],
        'description'  => ['required', 'string'],
        'price'        => ['required', 'numeric'],
        'is_featured'  => ['required'],
        'is_limited'   => ['required'],
        'stock'        => ['required', 'integer'],
        'category_id'  => ['required', 'exists:categories,id'],
        'images'       => ['sometimes', 'array'],
        'images.*'     => ['image', 'mimes:jpg,jpeg,png,webp', 'max:4096'],
    ]);
    // ... lanjut proses dengan $validated
}

Mengapa penting?

Mencegah Mass Assignment, Type Confusion, serta upload file berbahaya. Validasi gagal mengembalikan response 422 dengan pesan kesalahan yang dapat dirender langsung oleh frontend Inertia.

4.6 Database Transaction untuk Atomicity

Operasi yang menyentuh beberapa tabel sekaligus (mis. simpan produk + simpan gambar) dibungkus dalam DB::transaction() agar konsisten dan rollback otomatis ketika terjadi error:

DB::transaction(function () use ($request, $validated) {
    $product = Product::create([
        'title'        => $validated['title'],
        'description'  => $validated['description'],
        'price'        => $validated['price'],
        'stock'        => $validated['stock'],
        'is_limited'   => $validated['is_limited'] === 'true' ? 1 : 0,
        'is_featured'  => $validated['is_featured'] === 'true' ? 1 : 0,
        'category_id'  => $validated['category_id'],
    ]);

    if ($request->hasFile('images')) {
        foreach ($request->file('images') as $file) {
            $path = $file->store('uploads/products', 'public');
            $product->images()->create([
                'image_path' => $path,
                'is_primary' => 1,
            ]);
        }
    }
});

4.7 Role-Based Access Control (RBAC) via Middleware

Dengan bantuan spatie/laravel-permission, setiap rute admin dijaga oleh middleware permission:<nama>. Pendekatan ini mengikuti Principle of Least Privilege: pengguna hanya melihat & mengeksekusi aksi sesuai izinnya.

// routes/web.php
Route::middleware(['auth', 'verified'])->group(function () {

    Route::middleware(['permission:product.create'])->group(function () {
        Route::get('product/create',  [ProductController::class, 'create'])->name('productCreate');
        Route::post('product/store',  [ProductController::class, 'store'])->name('productStore');
    });

    Route::middleware(['permission:product.view'])->group(function () {
        Route::get('product',         [ProductController::class, 'index'])->name('product');
        Route::get('product/{id}',    [ProductController::class, 'show'])->name('productShow');
    });

    Route::middleware(['permission:product.edit'])->group(function () {
        Route::post('product/update/{id}', [ProductController::class, 'update'])->name('productUpdate');
    });

    Route::middleware(['permission:product.delete'])->group(function () {
        Route::delete('product/{id}', [ProductController::class, 'destroy'])->name('productDelete');
    });
});

Granularitas izin

Izin dibagi atas pola resource.action (mis. product.create, shipments.edit) sehingga peran (role) bisa dirakit secara dinamis dari kombinasi permission tanpa mengubah kode.

4.8 Service Layer untuk Bisnis Logic

Untuk logika non-trivial yang berpotensi dipakai ulang, kami mengekstrak ke Service class sehingga controller tetap ringkas dan logika dapat di-unit test. Contoh: App\Services\DeliveryService bertanggung jawab atas geocoding dan kalkulasi ongkir.

// app/Services/DeliveryService.php
class DeliveryService
{
    /**
     * Build a shipping quote for a destination address.
     * Geocodes the address, measures the distance from the store origin and
     * applies a pure per-km rate. Falls back to a flat fee when the address
     * cannot be located so checkout never breaks.
     */
    public function quote(string $address, ?string $city = null, /* ... */): array
    {
        $candidates = array_unique(array_filter([
            implode(', ', array_filter([$address, $village, $district, $city, $province, $postalCode])),
            implode(', ', array_filter([$village, $district, $city, $province])),
            // ... candidates progressively coarser
        ]));

        $coords = null;
        foreach ($candidates as $candidate) {
            $coords = $this->geocode(trim($candidate));
            if ($coords) break;
        }

        if (! $coords) {
            return [
                'distance_km'   => null,
                'shipping_cost' => (int) config('delivery.fallback_fee'),
                'geocoded'      => false,
            ];
        }

        $distance   = $this->haversine(/* origin */, $coords['lat'], $coords['lng']);
        $billableKm = max(1, (int) ceil($distance));
        $cost       = $billableKm * (int) config('delivery.rate_per_km');

        return [
            'distance_km'   => round($distance, 1),
            'shipping_cost' => $cost,
            'geocoded'      => true,
        ];
    }
}

Single Responsibility

Controller bertanggung jawab atas HTTP, sedangkan kalkulasi ongkir & geocoding tinggal di DeliveryService. Bila aturan ongkir berubah, hanya satu kelas yang perlu disentuh.

4.9 Caching Strategy

Layanan eksternal (Nominatim OpenStreetMap) memiliki rate limit dan latensi tinggi. Hasil geocoding sukses kami cache 30 hari, sedangkan kegagalan tidak di-cache supaya bisa retry:

$cacheKey = 'delivery:geocode:'.md5($query);

if ($cached = Cache::get($cacheKey)) {
    return $cached;
}

// ... call external service
if (! empty($data[0]['lat']) && ! empty($data[0]['lon'])) {
    $coords = ['lat' => (float) $data[0]['lat'], 'lng' => (float) $data[0]['lon']];
    Cache::put($cacheKey, $coords, now()->addDays(30));
    return $coords;
}

4.10 Type Safety dengan TypeScript

Seluruh frontend ditulis dalam TypeScript dengan tsc --noEmit sebagai gate kualitas. Tipe untuk props halaman didefinisikan terpusat di resources/js/types/ sehingga konsisten antar halaman. Wayfinder bahkan menghasilkan helper rute bertipe sehingga kesalahan nama rute terdeteksi sebelum runtime.

// resources/js/pages/dashboard.tsx
import { dashboard } from '@/routes';
import { type BreadcrumbItem } from '@/types';

const breadcrumbs: BreadcrumbItem[] = [
  { title: 'Dashboard', href: dashboard().url },

4.12 Defensive Programming & Fallback

Antarmuka yang menyentuh sistem eksternal selalu memiliki fallback agar pengalaman pengguna tidak terganggu jika layanan pihak ketiga gagal. Contoh pada DeliveryService::quote(): bila geocoding gagal, sistem mengembalikan fallback_fee dari konfigurasi, bukan melempar exception ke halaman checkout. Demikian pula pada controller produk, pemetaan gambar memiliki default jika koleksi kosong:

$images = $product->images->map(/* ... */)->toArray();

if (empty($images)) {
    $images = [
        '/assets/ProductDetail/productdetail-1.jpg',
        // fallback agar tampilan tetap valid
    ];
}
Catatan: Defensive programming dipakai khusus pada batas sistem (input pengguna, response service eksternal). Pada kode internal, kami percaya pada kontrak fungsi agar tidak menambah pemeriksaan yang tidak perlu (sesuai prinsip YAGNI).

BAB V — KESIMPULAN

Berdasarkan penerapan yang telah didokumentasikan pada bab-bab sebelumnya, dapat disimpulkan bahwa proyek Order Store mengadopsi prinsip penulisan kode yang sejalan dengan guidelines dan best practice industri:

  1. Struktur & konvensi: project mengikuti struktur direktori standar Laravel dan PSR-4 autoloading, membuat codebase mudah ditelusuri dan di-onboarding.
  2. Pemisahan tanggung jawab: arsitektur MVC dipertegas dengan Service Layer untuk logika kompleks (mis. DeliveryService), sehingga controller tetap fokus pada urusan HTTP.
  3. Validasi & keamanan: validasi diletakkan pada batas sistem (controller), parameter binding Eloquent meniadakan risiko SQL Injection, dan RBAC via middleware spatie/laravel-permission menerapkan Principle of Least Privilege.
  4. Konsistensi data: operasi multi-tabel dilindungi DB::transaction() sehingga atomicity terjaga.
  5. Performa & ketahanan: caching hasil geocoding dan fallback fee menjaga checkout tetap berfungsi meskipun layanan eksternal bermasalah.
  6. Kualitas kode otomatis: Pint, Prettier, ESLint, dan TypeScript --noEmit menjamin gaya penulisan seragam tanpa beban manual.
  7. Type-safety & reusability: TypeScript di frontend, komponen React modular, serta helper rute bertipe (Wayfinder) menurunkan jumlah bug regresi.

Penerapan prinsip-prinsip tersebut tidak hanya mempercantik kode, melainkan juga memberi dampak terukur: onboarding pengembang baru menjadi cepat, code review berfokus pada substansi (bukan format), regression berkurang, dan sistem siap dikembangkan ke arah scalability berikutnya seperti fitur audit log, notifikasi otomatis, ataupun aplikasi mobile.

Rekomendasi Pengembangan Lanjutan

  • Memindahkan validasi besar dari controller ke Form Request kelas tersendiri untuk meningkatkan keterujian dan keterbacaan.
  • Menambahkan Policy Laravel untuk otorisasi berbasis instance (mis. apakah courier X boleh mengubah shipment Y).
  • Menambah cakupan feature test Pest untuk alur checkout, webhook Midtrans, dan shipping quote.
  • Menerapkan commit hook (mis. via Husky atau lefthook) agar Pint, Prettier, ESLint, dan TypeScript dijalankan otomatis sebelum commit.
  • Menambah dokumentasi API contract (OpenAPI) untuk endpoint publik (get-all-products, get-filtered-products, dsb.).
